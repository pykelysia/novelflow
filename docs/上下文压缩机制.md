# NovelFlow 上下文压缩机制

NovelFlow 的上下文管理由两个相互配合的层组成：**SummarizationMiddleware**（运行时压缩）和 **SessionMiddleware**（持久化层）。两者均作为 `adk.ChatModelAgentMiddleware` 挂载在主 Agent 上，按固定顺序执行。

---

## 1. 中间件挂载顺序

```go
// agents/core/agent.go:70
config.Handlers = append(config.Handlers,
    &SafeToolMiddleware{},   // 1. 工具错误隔离
    summarizationMW,         // 2. 上下文压缩（Lite LLM）
    newSessionMiddleware(s), // 3. 消息持久化（MongoDB）
)
```

顺序关键点：SummarizationMiddleware 先于 SessionMiddleware 执行，因此 **压缩后的消息才会被写入 MongoDB**，避免历史无限膨胀。

Sub-Agent（outline / write / review）**不挂载** SummarizationMiddleware，只有主 Agent 有上下文压缩能力。

---

## 2. SummarizationMiddleware（运行时压缩）

**来源**：`github.com/cloudwego/eino/adk/middlewares/summarization`

**初始化**（`agents/core/agent.go:125-138`）：

```go
summarization.New(ctx, &summarization.Config{
    Model: cm,  // Lite LLM（lite_llm.* 配置）
    Trigger: &summarization.TriggerCondition{
        ContextTokens: 200000,
    },
})
```

**触发条件**：当前轮次传入模型的上下文 token 数超过 **200,000**。

**行为**：
- 触发后，用 Lite LLM 对历史消息做摘要，将多轮历史压缩为若干摘要消息，替换原始历史。
- 压缩在 `BeforeModel` 阶段发生，对当前轮次的模型调用立即生效。
- Lite LLM 配置独立于主模型，通过 `GetLiteChatModel` 读取 `lite_llm.*` 配置段。

## 3. SessionMiddleware（持久化层）

**来源**：`agents/core/session_middleware.go`

### 3.1 三个钩子

| 钩子 | 触发时机 | 操作 |
|---|---|---|
| `BeforeModelRewriteState` | 每轮第一次模型调用前 | 从 MongoDB 加载历史，注入 `state.Messages`；保存当前用户消息；记录 `savedLen` |
| `AfterModelRewriteState` | 模型响应后 | 将 `savedLen` 之后新增的消息追加写入 MongoDB |
| `AfterToolCallsRewriteState` | 一轮工具调用结束后 | 同上，保存工具调用和工具结果消息 |

### 3.2 历史注入逻辑

`BeforeModelRewriteState` 通过 run-local value `_nf_session_saved_len` 判断是否是本轮的第一次调用（同一轮内后续调用跳过注入）：

```
system messages
  ↓ [注入位置]
history messages (从 MongoDB 按 created_at 升序读取)
  ↓
current user message
```

### 3.3 增量保存

`persistNewMessages` 利用 `savedLenKey` 记录上次保存到 `state.Messages` 的长度，每次只写入新增片段，避免重复。

## 4. 消息类型与持久化策略

**定义**（`agents/session/message.go`）：

| 类型常量 | 说明 | 写入 MongoDB |
|---|---|---|
| `ContentType` | 普通对话（user / assistant） | ✅ |
| `ToolResultType` | 工具调用（assistant 发起）或工具结果（tool 角色） | ✅ |
| `ThinkingType` | 模型推理过程（reasoning_content） | ❌ 仅流式推送给调用方 |
| `ToolType` | 工具名称（流式事件标记，非消息） | ❌ 不持久化 |

`ThinkingType` 只在 `RunA` 流式输出阶段触发 `handlerFunc`，不经过 SessionMiddleware 的保存路径，因此**不会污染历史**，也不会占用下轮上下文 token。

---

## 5. 工具调用的序列化格式

MongoDB 中工具调用以 `ToolResultType` 类型存储，通过 `Role` 区分方向（`agents/core/session_middleware.go:107-116`）：

| 存储内容 | Role | Content 字段 | ToolResult 字段 |
|---|---|---|---|
| 工具调用发起 | `assistant` | `"<toolName>\n<arguments>"` | — |
| 工具调用结果 | `tool` | `"<toolName>"` | 结果内容 |

`Load()` 重建时自动生成 `call_0`, `call_1`... 格式的 `ToolCallID` 并关联前后消息，保证 OpenAI/Anthropic 协议合规。

---

## 6. 两层协作时序

```
用户输入
    │
    ▼
BeforeModelRewriteState (SessionMiddleware)
    └── 注入 MongoDB 历史 → 完整 state.Messages
    │
    ▼
[SummarizationMiddleware 内部 BeforeModel]
    └── token > 200k？→ 用 Lite LLM 压缩历史
    │
    ▼
模型调用（主 LLM）
    │
    ▼
AfterModelRewriteState (SessionMiddleware)
    └── 新增消息写入 MongoDB
    │
    ▼
工具调用执行（SafeToolMiddleware 包裹）
    │
    ▼
AfterToolCallsRewriteState (SessionMiddleware)
    └── 工具结果写入 MongoDB
    │
    ▼
（循环直到模型不再调用工具）
```

## 7. 配置参考

```yaml
# config.yaml
lite_llm:
  model_type: anthropic   # openai / anthropic / deepseek
  model_name: claude-haiku-4-5-20251001
  base_url: ""
  api_key: "..."
  max_tokens: 8192
```

压缩阈值（`200000` tokens）当前硬编码在 `agents/core/agent.go:buildSummarization`，如需调整直接修改 `TriggerCondition.ContextTokens`。
